## 3. 之前为什么会被认为“还不完全满足”

之前最大的争议点主要有两个。

### 3.1 Java 源码分析曾经主要依赖规则匹配

早期版本中，Java 类、方法和复杂度分析更多依赖：

1. 正则表达式
2. 关键字判断
3. 大括号层级跟踪

这种方式适合实验原型，但如果直接写成“已经完全按照 AST 解析实现”，说服力不够。

### 3.2 文档里把 AST 写成“计划使用”

之前一些文档中的表达还是：

1. 建议使用 JavaParser
2. 后续可升级为 ASTParser

这会让人看起来像是：

1. 功能做了
2. 但底层实现方式还没补齐

## 4. 现在补齐了什么

当前版本已经把 Java 结构分析改成了 **AST 优先** 的实现。

### 4.1 AST 解析已经落到代码里

后端新增了基于 JDK 编译器的 AST 解析支持：

1. [JavaAstSupport.java](/Users/xiazixuan/大三下/软件质量保证/exp/end/src/main/java/com/metriclab/service/JavaAstSupport.java:1)

它不是简单的正则匹配，而是利用 Java 编译器提供的抽象语法树能力去提取：

1. 类
2. 接口
3. 字段
4. 方法
5. 继承关系
6. 类型引用
7. 方法体中的控制分支

### 4.2 控制流复杂度已经改成 AST 优先

当前复杂度分析服务：

1. 优先使用 AST 识别方法
2. 优先使用 AST 统计 `if / for / while / case / catch / && / || / ?:` 等决策点，并避免对 `switch` 本身重复计数
3. 如果运行环境不支持编译器 AST，再回退到旧规则方案

对应文件：

1. [ComplexityAnalysisService.java](/Users/xiazixuan/大三下/软件质量保证/exp/end/src/main/java/com/metriclab/service/ComplexityAnalysisService.java:1)

### 4.3 面向对象度量已经改成 AST 优先

当前面向对象分析服务：

1. 优先使用 AST 提取类、字段、方法、继承关系和类型引用
2. 在 AST 基础上计算 `CBO / RFC / DIT / NOC / NOA / NOO / CS / WMC / LCOM`
3. 如果 AST 不可用，再回退到旧规则方案

对应文件：

1. [ObjectOrientedAnalysisService.java](/Users/xiazixuan/大三下/软件质量保证/exp/end/src/main/java/com/metriclab/service/ObjectOrientedAnalysisService.java:1)

### 4.4 文档表达也同步修正

现在项目文档中不再写成“只是计划使用 AST”，而是明确写成：

1. 当前源码结构分析已使用 AST 优先方案
2. 使用的是 JDK Compiler API AST
3. 后续如果需要，还可以继续换成 Eclipse JDT ASTParser

## 5. 关于“老师建议使用 Eclipse ASTParser”该怎么解释

这一点要注意表述。

老师原文里的意思是：

1. 对 Java 程序，建议利用 Eclipse 中的 `ASTParser` 得到 AST 再分析

这里的关键词是“建议”，不是“必须只允许 Eclipse JDT 这一种实现”。

所以更稳妥的说法是：

> 本项目已经采用 AST 解析思想来处理 Java 源码，当前实现基于 JDK Compiler API 提供的抽象语法树能力完成类、方法、字段和控制分支提取，满足了“基于 AST 进行源码结构分析”的实验目标；如果后续需要进一步增强对复杂语法的适配能力，还可以平滑替换为 Eclipse JDT ASTParser。

这样说有两个好处：

1. 不会把自己说死成“我必须就是 Eclipse 插件里的那个类”
2. 又能说明你已经真正按 AST 方式做了，而不是纯规则匹配
