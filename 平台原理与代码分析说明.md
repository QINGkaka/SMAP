# 自动化软件度量平台原理与代码分析说明

## 1. 文档目的

这份文档用于说明本项目与《软件质量保证》课程中“软件质量模型、软件测量与软件度量、软件规模度量、面向对象度量”等知识之间的对应关系，重点回答四个问题：

1. 课程里讲的软件质量、软件度量、软件质量保证到底是什么。
2. 本项目到底实现了哪些度量功能，分别对应哪些课件知识点。
3. 系统是如何对 Java 源代码、模型文件和用户输入进行分析的。
4. 每个指标的计算公式、实现思路和适用边界是什么。

这份文档既可以作为实验报告中的“系统原理说明”，也可以作为答辩时的技术说明底稿。

## 2. 课程背景与项目定位

### 2.1 软件质量保证课程的核心目标

从课程整体内容看，软件质量保证不是单一的“测试课”或者“代码规范课”，而是围绕软件质量开展的一整套系统性活动。课件中强调的软件质量保证包括：

1. 明确质量需求。
2. 建立质量模型。
3. 用度量方法理解、评价、预测、控制和改进软件。
4. 结合评审、静态分析、测试等手段发现问题。
5. 在软件生命周期中持续改进质量。

本项目的定位不是替代完整的软件测试平台，也不是替代 SonarQube 这类工业级代码质量平台，而是实现一个**面向课程实验的中小型软件度量自动化工具**，把课件中的主要度量知识落地到一个可运行、可展示、可导出的 Web 系统中。

### 2.2 本项目在课程体系中的位置

结合课程简介与前几章内容，本项目主要覆盖了以下教学目标：

1. 理解软件质量、质量需求与质量特性的关系。
2. 理解软件测量、度量、指标的区别。
3. 理解软件项目、软件过程、软件产品三类度量对象。
4. 掌握规模度量、复杂度度量、面向对象度量、功能点和用例点等核心方法。
5. 用自动化工具完成分析、统计、展示和报告导出。

因此，本项目属于“软件质量保证中的自动化度量工具实现”，重点在**静态分析、规模估算、结构评价和结果解释**。

## 3. 软件质量保证基础知识

### 3.1 软件质量、差错、缺陷、失效

课件中对软件问题术语作了严格区分：

1. `Mistake` / 差错：人的不正确行为。
2. `Defect` / 缺陷：差错在程序、设计、文档中的具体体现。
3. `Failure` / 失效：带缺陷的软件在运行时表现为错误结果或功能失常。

它们的关系可以概括为：

```text
人的差错 -> 软件缺陷 -> 运行失效
```

这也是软件质量保证要做的事：尽可能在缺陷进入运行阶段之前，通过规范、设计方法、静态分析和测试把问题提前发现出来。

### 3.2 软件质量的内在质量与外在质量

课程中把程序质量分为两个视角：

1. 外在质量：用户能直接感知的质量。
   - 正确性
   - 可靠性
   - 易用性
   - 响应速度
   - 功能满足程度
2. 内在质量：开发者和维护者更关心的质量。
   - 可读性
   - 可理解性
   - 可修改性
   - 可维护性
   - 可重用性

本项目主要分析的是**内在质量及其可量化表现**，例如：

1. 代码规模是否过大。
2. 方法控制逻辑是否过于复杂。
3. 类之间耦合是否过高。
4. 类内聚是否不足。
5. 设计模型是否过于庞杂。

这些内在质量指标会间接影响外在质量，例如高复杂度和高耦合通常会带来更高的缺陷率和更差的维护效率。

### 3.3 软件质量保证方法与本项目的关系

课件中把程序质量保证方法概括为：

1. 遵循编码规范。
2. 采用程序设计方法。
3. 进行代码重用。
4. 开展结对编程。
5. 进行人工审查。
6. 进行自动化静态分析。
7. 开展软件测试。

本项目直接实现的是第 6 类，即**自动化静态分析与度量分析**，并为第 5 类人工审查提供数据支持。例如：

1. 通过复杂度结果帮助定位需要人工复审的方法。
2. 通过 CBO、RFC、LCOM 帮助识别设计不合理的类。
3. 通过 LoC、功能点、用例点、估算结果支持项目级评估。

### 3.4 软件质量需求

课件中通常把质量需求与功能需求区分开来，并强调软件需求至少可以从以下三个方面理解：

1. 功能需求
2. 非功能需求
3. 性能需求

对本项目而言，可以对应为：

1. 功能需求  
   支持项目管理、文件上传、代码行度量、控制流复杂度度量、面向对象度量、模型文件度量、功能点、用例点、估算分析、智能分析与报告导出。
2. 非功能需求  
   结果口径一致、可重复分析、按项目隔离数据、支持文件级定位、界面可读、报告可导出。
3. 性能需求  
   面向课程实验规模的 Java 文件和模型文件，能够在较短时间内完成分析并返回结果。

## 4. 软件质量模型与软件度量基础

### 4.1 质量模型

课件中介绍了多个经典质量模型，包括：

1. `McCall` 质量模型
2. `Boehm` 质量模型
3. `ISO/IEC 9126`
4. `ISO/IEC 25010`

其中 `ISO/IEC 25010` 是现在最常用的质量特性框架之一，主要包括：

1. 功能适合性 `Functional Suitability`
2. 性能效率 `Performance Efficiency`
3. 兼容性 `Compatibility`
4. 易用性 `Usability`
5. 可靠性 `Reliability`
6. 安全性 `Security`
7. 可维护性 `Maintainability`
8. 可移植性 `Portability`

本项目并不是直接把这 8 个特性全部做成“单一分数”，而是通过若干基础度量间接支撑其中几项，尤其是：

1. 可维护性
2. 可理解性
3. 结构合理性
4. 规模可控性
5. 开发成本可预测性

### 4.2 课件中的典型质量度量公式

除结构和规模类度量外，课件在质量模型部分还强调了可靠性与可用性度量，典型公式包括：

1. `MTTF`：平均失效前时间
2. `MTTR`：平均修复时间
3. `MTBF`：平均故障间隔时间

它们之间的关系为：

```text
MTBF = MTTF + MTTR
```

可用性常用公式为：

```text
Availability = MTTF / (MTTF + MTTR) * 100%
```

这类指标用于运行期质量评价，需要长期运行日志、故障记录和修复时间数据。本项目当前输入主要是源代码、模型文件和人工录入参数，因此没有把 `MTTF/MTTR/MTBF/Availability` 做成直接可计算模块。文档保留这些公式，是为了和课件知识保持一致，并明确说明它们属于**运行期质量度量**，而本项目当前主要实现的是**静态结构与规模度量**。

### 4.3 测量、度量、指标的区别

根据课件，以下概念需要区分：

1. `Measure` / 测量值  
   对某个实体属性进行赋值后的原始结果。

2. `Metric` / 度量指标  
   对一个或多个测量值进行组合、解释后形成的指标。

3. `Indicator` / 指示器  
   用于支持判断、决策和管理的更高层解释结果。

在本系统中可以这样理解：

1. `LoC = 260` 是测量结果。
2. `某方法圈复杂度 = 5`、`某类 CBO = 4` 是度量指标。
3. `高风险类数量 = 1`、`建议优先重构 EnrollmentService` 是指标解释或指示结果。

### 4.4 度量对象

课件中强调，软件度量对象可以是：

1. 软件项目 `Project`
2. 软件过程 `Process`
3. 软件产品 `Product`

本项目当前主要分析：

1. 软件产品：代码、类、方法、模型文件。
2. 软件项目：规模、功能点、用例点、工作量、成本、开发周期。

对于软件过程的度量，例如缺陷注入率、评审效率、测试阶段缺陷密度等，本项目当前版本没有展开，因为这些需要过程数据、版本数据和缺陷流转数据，而不仅是源代码和模型文件。

### 4.5 量表类型

课件中介绍了软件测量常见量表：

1. 名义量表 `Nominal`
2. 顺序量表 `Ordinal`
3. 区间量表 `Interval`
4. 比率量表 `Ratio`
5. 绝对量表 `Absolute`

本系统中的对应例子如下：

| 量表类型 | 含义 | 本系统示例 |
|---|---|---|
| 名义量表 | 分类，不比较大小 | 文件类型、类类型、项目语言 |
| 顺序量表 | 可排序，不可精确比较倍数 | 风险等级 `LOW/MEDIUM/HIGH` |
| 区间量表 | 差值有意义，但比例关系不稳定 | 课件理论知识点，本系统无单独代表性模块 |
| 比率量表 | 有绝对零点，可比较倍数 | LoC、复杂度、RFC、FP、UCP、KLOC、成本 |
| 绝对量表 | 直接计数 | 文件数、类数、接口数、继承关系数 |

## 5. 系统总体架构与数据流

### 5.1 架构概览

本系统采用前后端分离架构：

1. 前端：`Vue`
   - 负责页面展示、文件上传、参数录入、结果可视化、导出。
2. 后端：`Java + Spring Boot`
   - 负责文件解析、指标计算、结果持久化、报告生成。

### 5.2 存储方式

系统不依赖数据库，采用本地文件存储：

1. `metric-data/projects.json`  
   保存项目基础信息。
2. `metric-data/projects/{projectId}/uploads/`  
   保存上传文件。
3. `metric-data/projects/{projectId}/tasks/`  
   保存各类分析结果。
4. `metric-data/projects/{projectId}/reports/`  
   保存导出的 Markdown、XML 等报告。

这种方式适合课程实验：

1. 部署简单。
2. 可直接检查原始输入与分析结果。
3. 不依赖额外数据库环境。

### 5.3 系统分析流程

平台整体处理流程如下：

```text
新建项目
-> 上传 Java / ZIP / XML / XMI / OOM 文件
-> 选择度量模块
-> 按项目整体或按指定文件分析
-> 保存结果
-> 页面展示
-> 导出报告
```

### 5.4 分析范围

当前多个模块支持两种分析范围：

1. 整个项目
2. 指定一个或多个文件

这让系统既能做项目级评价，也能做局部文件定位分析，符合课件中“理解、评价、控制和改进”的思想。

## 6. 输入实体与分析对象

指导书中要求系统能够处理类图、流程图、用例图、源代码、用户输入等实体。本系统采用如下实现方式：

### 6.1 源代码输入

支持：

1. `.java`
2. `.zip` 中的 Java 源文件

主要用于：

1. 代码行度量
2. 控制流复杂度度量
3. 面向对象度量

### 6.2 模型文件输入

支持：

1. `.xml`
2. `.xmi`
3. `.oom`

这些文件被视为**结构化模型文件**，用于提取：

1. 类
2. 接口
3. 属性
4. 操作
5. 继承关系

这比 `png/jpg` 这类静态图片更适合自动化度量，因为图片本身不包含可直接计算的 UML 结构语义。

### 6.3 用户输入

用于录入：

1. 功能点明细
2. 用例点明细
3. 技术复杂度因子
4. 环境复杂度因子
5. 估算参数

这部分对应课件中“并非所有度量都能仅从代码自动推导”的现实情况。

## 7. 软件规模度量

### 7.1 代码行度量 LoC

#### 7.1.1 理论基础

课件中把软件规模度量视为软件项目评估的重要基础。规模可以通过多种方式反映：

1. 代码长度
2. 功能规模
3. 模块数量
4. 复杂度
5. 可重用程度

其中 `LoC` / `KLoC` 是最基础、最直接的代码长度度量方式。

#### 7.1.2 本系统统计内容

系统统计：

1. Java 文件数
2. 总行数 `Total Lines`
3. 有效代码行 `SLoC`
4. 注释行
5. 空行
6. 注释率

#### 7.1.3 实现方式

对应后端模块：

1. `LocAnalysisService`

处理逻辑：

1. 读取项目上传的 Java 文件或 ZIP 中的 Java 文件。
2. 过滤 `__MACOSX/`、`._*`、`.DS_Store` 等无效条目。
3. 逐行扫描代码。
4. 区分空行、注释行、有效代码行。
5. 汇总项目级统计结果。

#### 7.1.4 指标意义

LoC 不能单独代表质量，但它是许多后续分析的基础：

1. 规模越大，通常维护成本越高。
2. 可与复杂度结合判断“大而复杂”的高风险模块。
3. 可为 COCOMO 等估算模型提供输入。

### 7.2 功能规模度量与代码规模的关系

课件中强调，LoC 只反映代码长度，不等于功能规模。因此系统进一步实现了：

1. 功能点 `FP`
2. 用例点 `UCP`

这两类指标用于从功能视角估计软件规模，与 LoC 形成互补。

## 8. 控制流复杂度度量

### 8.1 圈复杂度理论

课件中传统复杂度度量的核心指标之一是圈复杂度 `Cyclomatic Complexity`。它反映方法控制流中独立路径的数量。工程上常用的简化计算方式是：

```text
CC = 1 + 决策点数量
```

常见决策点包括：

1. `if`
2. `else if`
3. `for`
4. `while`
5. `do while`
6. `case`
7. `catch`
8. 三目运算符 `?:`
9. 短路逻辑 `&&`、`||`

### 8.2 复杂度与质量的关系

复杂度高通常意味着：

1. 方法不容易理解。
2. 测试路径增多。
3. 更容易埋藏边界错误。
4. 修改后更容易引入回归缺陷。

因此复杂度度量直接支撑可维护性分析和测试优先级排序。

### 8.3 本系统如何实现

对应后端模块：

1. `ComplexityAnalysisService`
2. `JavaAstSupport`

当前实现采用**AST 优先静态分析**：

1. 读取 Java 源代码。
2. 使用 JDK Compiler API 构建 Java AST。
3. 遍历方法节点。
4. 识别控制结构。
5. 计算每个方法的圈复杂度。
6. 汇总文件级、项目级复杂度结果。

需要说明的是，实验指导书建议可以使用 Eclipse JDT 的 `ASTParser`。本系统并未直接使用 Eclipse JDT，而是使用 **JDK 自带的 Java Compiler API** 构建 AST，达到同类的静态结构分析目的。也就是说，本系统满足“基于 AST 进行静态分析”的技术思路，但具体实现不是 Eclipse 专有的 `ASTParser`。

### 8.4 本模块产出

系统输出：

1. 每个方法的复杂度值
2. 每个文件的平均复杂度
3. 项目级平均复杂度
4. 风险等级
5. 高复杂度方法列表

### 8.5 与课件知识的对应关系

本模块直接对应：

1. 传统复杂度度量
2. 程序控制结构分析
3. 静态分析发现复杂模块
4. 用复杂度支持测试与维护决策

## 9. 面向对象度量

### 9.1 面向对象程序特点

课件中指出，面向对象程序具有：

1. 抽象
2. 封装
3. 继承
4. 多态
5. 局部化

因此对面向对象程序的度量不能只看代码行数，还需要分析：

1. 类内方法规模
2. 类之间关系
3. 继承层次
4. 调用范围
5. 内聚和耦合

### 9.2 传统度量、CK、LK

课件中介绍了三类主要方法：

1. 传统度量：如方法复杂度、类或方法 LoC。
2. `CK` 度量族。
3. `LK` 度量族。

本项目当前重点实现：

1. 传统度量中的方法复杂度与代码行数支持
2. CK 六个核心指标
3. LK 中与类规模相关的若干指标

### 9.3 CK 指标

#### 9.3.1 WMC

`WMC`，`Weighted Methods per Class`，每个类的方法加权复杂度和。

课件中指出：

1. 若所有方法权重视为 1，则 WMC 等于方法数。
2. 若使用方法复杂度作为权重，则 WMC 能更真实反映类的实现复杂性。

本系统采用第二种方式：

```text
WMC = 类中各方法圈复杂度之和
```

#### 9.3.2 RFC

`RFC`，`Response for a Class`。

课件定义：

```text
RFC = |RS|
RS = {M} ∪ 所有方法直接调用的方法集合
```

含义是：当向一个类发送消息时，可能被执行的方法集合大小。RFC 越大，说明：

1. 类行为越复杂。
2. 理解和测试成本越高。
3. 变化影响面越大。

本系统在实现中统计：

1. 类自身方法
2. 类方法直接调用的方法

并排除构造器中的 `super(...)` / `this(...)` 链式调用，避免错误放大 RFC。

#### 9.3.3 DIT

`DIT`，`Depth of Inheritance Tree`。

定义：

```text
DIT = 从当前类到继承根的最大路径深度
```

意义：

1. DIT 大，说明继承层次深。
2. 可能提高复用能力。
3. 也可能提高理解和测试难度。

#### 9.3.4 NOC

`NOC`，`Number of Children`。

定义：

```text
NOC = 某类的直接子类数量
```

意义：

1. NOC 大，说明该类复用广。
2. 也意味着父类变化影响范围大。

#### 9.3.5 CBO

`CBO`，`Coupling Between Object Classes`。

课件特别强调：`CBO` 度量的是类之间耦合，**通常不把继承关系本身计入耦合值**。

本系统实现时遵循这一口径：

1. 字段类型引用计入耦合
2. 方法参数和返回值中的真实依赖计入耦合
3. 方法体中对其他类的显式类型使用计入耦合
4. `extends`、`implements` 不计入 CBO

这也是为什么样例中 `GraduateStudent` 仅因继承 `Student` 而不被判定为有额外 CBO。

#### 9.3.6 LCOM

`LCOM`，`Lack of Cohesion in Methods`。

课件给出的经典思路是：

1. 设类中方法对属性的访问集合分别为 `I1, I2, ..., In`
2. 若两个方法访问的属性集合无交集，则记入 `P`
3. 若有交集，则记入 `Q`
4. 最终：

```text
LCOM = max(P - Q, 0)
```

LCOM 越大，说明：

1. 类中方法彼此关联较弱。
2. 类承担了过多不相关职责。
3. 类应考虑拆分。

本系统按这个经典口径实现。

### 9.4 LK 类规模相关指标

结合实验要求和当前实现，本系统补充统计：

1. `NOA`：属性数
2. `NOO`：操作数
3. `CS`：类规模表征

这些指标用于从类结构规模视角补充 CK 分析结果。

### 9.5 设计阶段与实现阶段的对应

课件中强调：

1. `WMC`、`DIT`、`NOC`、`CBO` 可用于设计阶段类图分析。
2. `RFC`、`LCOM` 更依赖实现阶段的类代码分析。

本项目同时支持：

1. 代码层分析：通过 Java AST 计算 CK/LK
2. 模型层分析：通过 UML 风格模型文件统计类、属性、操作、继承关系

因此基本符合课件中“设计阶段 + 实现阶段联合度量”的思路。

### 9.6 本系统如何实现

对应后端模块：

1. `ObjectOrientedAnalysisService`
2. `JavaAstSupport`

处理逻辑：

1. 解析类、接口、字段、方法。
2. 识别继承关系。
3. 统计方法复杂度。
4. 识别方法调用关系。
5. 识别类型引用关系。
6. 基于公式计算 CK/LK 指标。

## 11. 功能点度量

### 11.1 理论基础

课件中介绍了功能点度量的多种体系，如：

1. `IFPUG`
2. `Mk II`
3. `COSMIC`
4. `NESMA`
5. `FiSMA`

本系统当前实现的是最常见、也最适合课程实验落地的 `IFPUG` 功能点方法。

### 11.2 IFPUG 的五类功能

课件把功能点分为两类、五项：

1. 数据功能
   - `ILF`：内部逻辑文件
   - `EIF`：外部接口文件
2. 事务功能
   - `EI`：外部输入
   - `EO`：外部输出
   - `EQ`：外部查询

### 11.3 复杂度判定

按照课件口径：

1. 事务功能复杂度由 `DER/FTR` 或 `DET/FTR` 类型规则判定
2. 数据功能复杂度由 `DET/RET` 判定

本系统当前实现采用：

1. `EI / EO / EQ` 按 `DET + FTR` 自动判级
2. `ILF / EIF` 按 `DET + RET` 自动判级

### 11.4 UFP、VAF、FP

功能点计算过程为：

1. 先根据复杂度和权重求未调整功能点：

```text
UFP = 五类功能项加权求和
```

2. 再根据 14 个通用系统特性求调整因子：

```text
VAF = 0.65 + 0.01 * DI
DI = 14 个特性打分之和
```

3. 最终功能点：

```text
FP = UFP * VAF
```

### 11.5 本系统如何实现

对应后端模块：

1. `FunctionPointService`

系统支持两种输入模式：

1. `DETAILED`
   - 录入每项功能的详细 DET/FTR/RET 信息
   - 系统自动判断低、中、高复杂度
2. `ESTIMATED`
   - 直接录入低、中、高数量

### 11.6 结果意义

功能点相对 LoC 的优势在于：

1. 更接近用户可见功能规模。
2. 与编程语言无关。
3. 可用于项目早期估算。

课件中还给出了语言换算示例，例如 Java 每功能点约对应一定数量的 SLOC。本系统估算模块默认使用：

```text
Java ≈ 60 SLOC / FP
```

用于在缺少 KLOC 手工输入时，从 FP 推导规模量。

## 12. 用例点度量

### 12.1 理论基础

课件中介绍了 `Use Case Point`，即用例点方法。它由 Gustav Karner 提出，适合在需求或分析设计阶段根据参与者与用例规模进行估算。

### 12.2 基本构成

用例点的核心构成为：

1. `UAW`：未调整参与者权重
2. `UUCW`：未调整用例权重
3. `UUCP = UAW + UUCW`

#### 12.2.1 参与者权重

课件常用权重为：

1. 简单参与者 = 1
2. 一般参与者 = 2
3. 复杂参与者 = 3

#### 12.2.2 用例权重

按用例复杂度划分简单、一般、复杂，并给予不同权重。

### 12.3 技术复杂度因子与环境复杂度因子

课件给出：

```text
TCF = 0.6 + 0.01 * TFactor
ECF = 1.4 - 0.03 * EFactor
```

最终：

```text
UCP = UUCP * TCF * ECF
```

### 12.4 本系统如何实现

对应后端模块：

1. `UseCasePointService`

系统允许用户录入：

1. 参与者数量与复杂度分布
2. 用例数量与复杂度分布
3. 技术复杂度因子
4. 环境复杂度因子

然后自动计算：

1. `UAW`
2. `UUCW`
3. `UUCP`
4. `TCF`
5. `ECF`
6. `UCP`

### 12.5 作用

用例点适合项目早期估算：

1. 还没有完整代码时也能估算规模。
2. 比 LoC 更早可用。
3. 比纯经验估计更有结构化依据。

## 13. 工作量、成本、开发周期与人员估算

### 13.1 理论定位

指导书要求系统度量的软件实体包括：

1. 软件规模
2. 工作量
3. 成本
4. 开发时间
5. 人员

本系统使用 `Basic COCOMO` 做估算实现。

### 13.2 Basic COCOMO

COCOMO 的基本思想是根据软件规模 `KLOC` 估算工作量和开发周期。基础形式可表示为：

```text
Effort = a * (KLOC)^b
Schedule = c * (Effort)^d
Average Staff = Effort / Schedule
Cost = Effort * 人月成本
```

### 13.3 本系统如何实现

对应后端模块：

1. `EstimationService`

输入来源有三种：

1. 用户手工输入 `KLOC`
2. 由最新 LoC 分析结果换算得到 `KLOC`
3. 若无 LoC 但已有 FP 结果，则按 `Java 60 SLOC/FP` 估算 `KLOC`

系统输出：

1. `KLOC`
2. 工作量 `Effort`
3. 开发周期 `Schedule`
4. 平均人员 `Average Staff`
5. 成本 `Cost`

### 13.4 指标意义

这一模块实现了从“度量”到“预测”的过渡，即：

1. 先测规模
2. 再估工作量
3. 再估周期和成本

这与课件中“度量可用于理解、预测、评价、控制与改进”的思想一致。

## 15. Java 代码是如何被分析的

### 15.1 AST 优先静态分析

实验指导书建议：

> 对于 Java 程序，可以利用 Eclipse 中的 ASTParser 得到程序代码的 AST，再进行统计分析。

本系统采用的实现路线是：

1. 读取 Java 源代码
2. 使用 JDK Compiler API 构建 AST
3. 遍历语法树节点
4. 提取类、方法、字段、继承、调用、控制结构等信息
5. 基于 AST 结果进行复杂度和面向对象指标计算

因此，准确的表述应为：

> 本系统采用 AST 优先的静态分析方法对 Java 代码进行分析，实现上未直接使用 Eclipse JDT 的 ASTParser，而是使用 JDK 自带的 Java Compiler API 构建抽象语法树，并在此基础上完成复杂度与 CK/LK 指标统计。

### 15.2 AST 分析在系统中的体现

AST 分析主要体现在两个功能模块中：

1. 控制流图度量
2. 面向对象度量

它们分别利用 AST 获取：

1. 方法定义
2. 控制分支
3. 循环结构
4. `case`
5. `catch`
6. 三目与逻辑运算
7. 字段
8. 方法调用
9. 类型引用
10. 继承关系

### 15.3 代码级处理的关键点

为了保证结果口径正确，系统还做了若干细化处理：

1. ZIP 过滤 `__MACOSX/`、`._*`、`.DS_Store`
2. `switch` 不重复计数
3. `do while` 不重复拆分
4. `super(...)` / `this(...)` 不错误计入 RFC
5. 继承关系不错误计入 CBO

这些处理直接影响指标是否与课件定义保持一致。

从课程知识视角看，本项目的核心价值不只是“把几个数字算出来”，而是把课件中的软件度量知识组织成了一个完整的自动化工具链：
