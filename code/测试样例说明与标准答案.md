# 测试样例说明与标准答案

这份目录用于给你的软件度量平台做回归测试。下面的组织顺序，已经按你前端左侧菜单从上到下排列，方便你逐项对照。

## 1. 测试目录说明

- `java-metric-sample/`
  - 一组专门用于测试 `LoC / 圈复杂度 / CK / LK` 的 Java 文件
- `model-files/academic-model.xml`
  - 一组用于测试 `.xml` 模型文件解析的 UML 风格结构化文件
- `model-files/academic-model.xmi`
  - 一组用于测试 `.xmi` 模型文件解析的 UML 风格结构化文件
- `model-files/academic-model.oom`
  - 一组用于测试 `.oom` 模型文件解析的 XML 风格结构化文件

建议做法：

1. Java 样例单独建一个项目上传。
2. 三个模型文件可分别单独建项目上传，便于核对单文件答案。
3. 功能点、用例点、估算分析按本文后半部分的手工输入验证。
4. 不要把 Java 样例和模型文件全部混在一个项目里做第一次验算，否则结果会叠加，不方便比对。

## 3. 代码行度量

Java 测试项目路径：

`code/java-metric-sample/src/main/java/sample/metrics`

### 3.1 样例设计目的

这组样例故意分成不同风格的类：

1. `Person`
   - 简单类
   - 用于测试基础方法统计和低复杂度
2. `Student`
   - 有继承、有属性引用、有方法调用
   - 用于测试 `CBO / RFC / DIT / NOC / WMC / LCOM`
3. `GraduateStudent`
   - 深一层继承
   - 用于测试 `DIT`
4. `EnrollmentService`
   - 耦合较多、方法复杂度较高、响应集较大
   - 用于测试高复杂类
5. `SplitStorage`
   - 专门用来测试 `LCOM > 0`
6. `ScoreStrategy`
   - 接口
   - 用于测试接口识别

如果不手工输入 `KLOC`，而是先得到第 7 节中的 `AFP = 32`，那么当前系统可按课件中的：

`Java 60 SLOC / FP`

`KLOC = 32 × 60 / 1000 = 1.92`

如果估算模块读取到的是这个换算值，则 `ORGANIC` 模式下预期结果应与 `KLOC = 1.92` 一致。

## 10. 智能分析

智能分析不是固定唯一数值答案，它更适合做“结果是否合理”的校验。

### 10.1 建议核对点

如果你已经先完成：

1. 代码行度量
2. 控制流图度量
3. 面向对象度量
4. 功能点 / 用例点 / 估算分析

那么智能分析里至少应该能体现这些趋势：

- `EnrollmentService` 应更容易被识别为重点风险类
- `SplitStorage` 应体现出内聚性问题
- `GraduateStudent` 应体现继承层次较深
- 如果做了估算，报告中应能引用规模、工期、成本或人员相关信息

### 10.2 验证方式

这一项主要看：

1. 能否正常生成分析结果
2. 结果中有没有结合前面各模块的数据
3. 风险判断和改进建议是否与前面模块结果一致

## 12. 使用提醒

1. 旧结果不会自动刷新，修改代码后要重新点击分析。
2. 模型文件如果多个一起上传，摘要结果会累加。
3. 功能点、用例点、估算分析依赖用户输入，不是从代码里自动推导。
4. 现在控制流、面向对象、模型文件度量都支持“整个项目”和“指定文件”两种分析方式，核对标准答案时要注意你当前使用的是哪一种模式。
